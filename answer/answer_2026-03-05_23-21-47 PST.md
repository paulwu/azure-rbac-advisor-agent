## Original Prompt

I'm deploying a three-tier web application into a Workload Landing Zone on Azure. The workload uses: Azure Virtual Network (spoke) with subnets and NSGs, Azure App Service, Azure SQL Database, Azure Storage Account (Blob + Queue), Azure Key Vault, Azure Container Registry, Azure Load Balancer and Application Gateway. I need RBAC role assignments for FOUR principals: (1) DevOps CI/CD pipeline SP, (2) Application Managed Identity, (3) Developer (read-only), (4) Database Administrator (SQL only).

---

## Principal 1 — DevOps CI/CD Pipeline (Service Principal)

> This principal provisions, configures, and deploys to all workload resources. It needs **management-plane** roles to create/update resources, plus **data-plane** roles for image push and secret management.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Spoke Virtual Network | `Network Contributor` | Resource Group | Management | Creates VNet, subnets, peerings, NSG associations, Private Endpoints. |
| Network Security Groups | `Network Contributor` | Resource Group | Management | Covered by VNet role — same `Network Contributor` at RG scope. |
| App Service Plan | `Web Plan Contributor` | Resource Group | Management | Creates/scales the App Service Plan. |
| App Service (Web Apps) | `Website Contributor` | Resource Group | Management | Creates apps, configures settings, deploys code, manages slots. |
| App Service — VNet Integration | `Network Contributor` | Spoke VNet | Management | Required in addition to `Website Contributor` for VNet Integration setup. |
| Azure SQL Database — Server | `SQL Server Contributor` | Resource Group | Management | Creates logical server, configures firewall, sets Entra ID admin. |
| Azure SQL Database — Database | `SQL DB Contributor` | Resource Group | Management | Creates/modifies databases. Does **not** grant data access. |
| Azure Storage Account | `Storage Account Contributor` | Resource Group | Management | Creates account, configures network rules, lifecycle policies. |
| Azure Key Vault — Resource | `Key Vault Contributor` | Resource Group | Management | Creates vault, configures network rules. Does **not** access secrets. |
| Azure Key Vault — Secrets | `Key Vault Secrets Officer` | Key Vault | Data | Deploys/rotates application secrets via pipeline. |
| Azure Key Vault — Certificates | `Key Vault Certificates Officer` | Key Vault | Data | Manages TLS certificates for App Service and App Gateway. |
| Azure Container Registry — Resource | `Contributor` | Registry | Management | Creates/configures the registry (no narrower built-in role exists). |
| Azure Container Registry — Images | `AcrPush` | Registry | Data | Pushes built container images. Includes pull. |
| Azure Load Balancer | `Network Contributor` | Resource Group | Management | Covered by VNet `Network Contributor` at RG scope. |
| Application Gateway | `Network Contributor` | Resource Group | Management | Covered by VNet `Network Contributor` at RG scope. |
| Diagnostics (all resources) | `Monitoring Contributor` | Resource Group | Management | Configures Diagnostic Settings across all resources. |

### Scope Recommendation

Assign `Network Contributor`, `Web Plan Contributor`, `Website Contributor`, `SQL Server Contributor`, `SQL DB Contributor`, `Storage Account Contributor`, `Key Vault Contributor`, `Contributor` (ACR), and `Monitoring Contributor` at the **Resource Group** scope. Assign `Key Vault Secrets Officer`, `Key Vault Certificates Officer` at the **Key Vault** resource scope. Assign `AcrPush` at the **Registry** resource scope.

## Principal 2 — Application Managed Identity (Runtime)

> This principal runs inside App Service and needs **data-plane** access to dependent resources at runtime. It should have **zero management-plane roles**.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Azure Key Vault | `Key Vault Secrets User` | Key Vault | Data | Reads secret values via Key Vault References in App Service settings. |
| Azure Storage — Blob | `Storage Blob Data Contributor` | Storage Account (or container) | Data | Reads and writes media blobs. Scope to specific container for narrowest access. |
| Azure Storage — Queue (send) | `Storage Queue Data Message Sender` | Queue | Data | Enqueues async job messages. |
| Azure Storage — Queue (receive) | `Storage Queue Data Message Processor` | Queue | Data | Dequeues and deletes processed messages. |
| Azure SQL Database | N/A — use SQL data-plane | Database | Data (SQL) | Azure RBAC does **not** grant SQL data access. Create a contained database user: `CREATE USER [app-identity] FROM EXTERNAL PROVIDER; ALTER ROLE db_datareader ADD MEMBER [app-identity]; ALTER ROLE db_datawriter ADD MEMBER [app-identity];` |
| Azure Container Registry | `AcrPull` | Registry | Data | Pulls container images for containerized App Service deployments. |

### Scope Recommendation

All roles at the **individual resource** scope (Key Vault, Storage Account/container/queue, Registry). For Storage, scope Blob and Queue roles to their respective containers/queues when possible. SQL access is configured inside the database via T-SQL, not Azure RBAC.

### Key Data Plane Note

If the app only needs to **enqueue** messages (producer pattern), assign only `Storage Queue Data Message Sender`. If it only needs to **dequeue** (consumer pattern), assign only `Storage Queue Data Message Processor`. Assign both only if the app does both.

## Principal 3 — Developer (Read-Only, No Secret Access)

> This principal can view resource configurations and read application logs, but must **not** modify production resources or access secret values.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| All resources (general) | `Reader` | Resource Group | Management | View-only access to all resource configurations, metadata, and metrics. |
| App Service — Log Stream | — | — | — | `Reader` alone cannot stream app logs. `Website Contributor` is needed for Log Stream, but that grants modification rights. Use Log Analytics queries instead. |
| Azure Key Vault | `Key Vault Reader` | Key Vault | Data (metadata) | Lists secret/key/certificate **names and metadata only** — cannot read secret **values**. |
| Azure SQL Database | `Reader` | SQL Server | Management | Views server/database configuration. No SQL data access. |
| NSG / VNet | `Reader` | Resource Group | Management | Can view effective security rules via `Reader` on NIC and NSG. |
| Log Analytics (if configured) | `Log Analytics Reader` | Workspace | Data | Queries application and diagnostic logs without modifying the workspace. |

### Scope Recommendation

Assign `Reader` at the **Resource Group** scope. Assign `Key Vault Reader` at the **Key Vault** resource scope. Assign `Log Analytics Reader` at the **Log Analytics Workspace** scope (platform landing zone) for log access.

### Why NOT `Key Vault Secrets User`

`Key Vault Secrets User` can read secret **values** — this violates the requirement that the developer must not access secret values. `Key Vault Reader` shows only metadata (secret names, expiry, enabled/disabled status).

## Principal 4 — Database Administrator (SQL Only)

> This principal has full control of Azure SQL Database resources but **no access** to any other resource in the workload.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Azure SQL — Server | `SQL Server Contributor` | SQL Server resource | Management | Full server management: firewall rules, Entra ID admin, failover groups, geo-replication. |
| Azure SQL — Databases | `SQL DB Contributor` | SQL Server resource | Management | Full database management: create/modify/delete databases, change tier/SKU. |
| Azure SQL — Security | `SQL Security Manager` | SQL Server resource | Management | Configure auditing, threat detection, vulnerability assessment. |
| Azure SQL — Data Access | N/A — use SQL data-plane | Database | Data (SQL) | Grant `db_owner` inside each database via T-SQL: `CREATE USER [dba-name] FROM EXTERNAL PROVIDER; ALTER ROLE db_owner ADD MEMBER [dba-name];` |

### Scope Recommendation

All Azure RBAC roles scoped to the **SQL Server resource** only — not at Resource Group level. This ensures the DBA cannot see or modify any other workload resources.

### Management vs. Data Plane for SQL

Azure RBAC (`SQL Server Contributor`, `SQL DB Contributor`) controls **resource management** (create servers, change tiers, configure firewall). It does **not** grant any SQL data access. Full database control (DDL + DML) requires `db_owner` role membership assigned via T-SQL inside the database by the Entra ID admin.

## Management Plane vs. Data Plane Summary

| Resource | Management Plane Role | Data Plane Role | Separation |
|---|---|---|---|
| Azure Key Vault | `Key Vault Contributor` | `Key Vault Secrets User` / `Key Vault Secrets Officer` | ✅ Full separation — management role cannot access secrets |
| Azure Storage Account | `Storage Account Contributor` | `Storage Blob Data Contributor` / Queue roles | ✅ Full separation — but `Storage Account Contributor` can list keys (equivalent to full data access) |
| Azure SQL Database | `SQL Server Contributor` / `SQL DB Contributor` | SQL T-SQL roles (`db_datareader`, etc.) | ✅ Full separation — Azure RBAC grants zero SQL data access |
| Azure Container Registry | `Contributor` | `AcrPush` / `AcrPull` | ✅ Full separation |
| App Service | `Website Contributor` | N/A (app settings are management plane) | ⚠️ `Website Contributor` can view app settings which may contain secrets — use Key Vault References |
| VNet / NSG / LB / AppGW | `Network Contributor` | N/A (no data plane) | N/A — management plane only |

---

## Notes

1. **`Storage Account Contributor` can list storage account keys** — this is functionally equivalent to full data-plane access. For the CI/CD pipeline, this is acceptable; for developers, `Reader` is safer.

2. **Azure SQL data access is entirely outside Azure RBAC** — all principals needing SQL data access must be added as contained database users via T-SQL by the Entra ID admin.

3. **`Website Contributor` can view app settings** — if secrets are stored in plain text app settings, any `Website Contributor` can read them. Always use **Key Vault References** (`@Microsoft.KeyVault(SecretUri=...)`) to prevent this.

4. **No narrower ACR management role exists** — `Contributor` scoped to the Registry is the minimum for creating/configuring the Container Registry.

5. **Disable API key / admin authentication** in production on: Key Vault, Storage Account, Container Registry, and SQL Server. Use Entra ID token-based auth exclusively.

6. **Private Endpoints** for all data services (SQL, Storage, Key Vault, ACR) require `Network Contributor` on the VNet subnet — this is already covered by the CI/CD pipeline's RG-scoped `Network Contributor`.
