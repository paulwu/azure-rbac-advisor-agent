## Scenario

A three-tier web application deployed into a **Workload Landing Zone** with: Spoke VNet + NSGs (networking), App Service (frontend + API), Azure SQL Database (backend), Azure Storage (Blob + Queue), Key Vault (secrets/certs), Container Registry (images), Load Balancer, and Application Gateway (traffic routing).

RBAC is structured for four principals with strict separation of duties.

---

## Principal 1 — DevOps CI/CD Pipeline (Service Principal)

> Deploys and configures all resources, pushes images to ACR, and manages App Service deployments.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Spoke VNet + Subnets + NSGs | `Network Contributor` | Resource Group | Management | Covers VNet, subnet, NSG, Load Balancer, and Application Gateway operations. |
| App Service Plan | `Web Plan Contributor` | Resource Group | Management | Creates and scales the compute plan. |
| App Service (Web App) | `Website Contributor` | Resource Group | Management | Deploys code, configures app settings, manages deployment slots. |
| Azure SQL Logical Server | `SQL Server Contributor` | Resource Group | Management | Creates server, configures firewall rules, sets Entra ID admin. Does NOT grant SQL data access. |
| Azure SQL Database | `SQL DB Contributor` | SQL Server resource | Management | Creates databases, changes tier/SKU. Does NOT grant SQL data access. |
| Azure Storage Account | `Storage Account Contributor` | Resource Group | Management | Creates account, configures network rules, lifecycle policies. Does NOT grant data access. |
| Azure Key Vault | `Key Vault Contributor` | Resource Group | Management | Creates vault, configures network/firewall rules. Does NOT grant access to secrets/keys/certs. |
| Azure Key Vault (secrets) | `Key Vault Secrets Officer` | Key Vault | Data | Deploy and rotate application secrets via pipeline. |
| Azure Container Registry | `Contributor` | Resource Group | Management | No ACR-specific management role exists; `Contributor` scoped to RG is the minimum for registry creation. |
| Azure Container Registry (images) | `AcrPush` | Registry | Data | Push (and pull) container images from CI/CD pipeline. |
| Load Balancer | `Network Contributor` | Resource Group | Management | Already covered by VNet assignment above. |
| Application Gateway | `Network Contributor` | Resource Group | Management | Already covered by VNet assignment above. |
| Diagnostic Settings (all resources) | `Monitoring Contributor` | Resource Group | Management | Configure log/metric shipping to Log Analytics. |

**Consolidated minimum for CI/CD SP at Resource Group scope:**
- `Network Contributor`
- `Web Plan Contributor`
- `Website Contributor`
- `SQL Server Contributor`
- `SQL DB Contributor`
- `Storage Account Contributor`
- `Key Vault Contributor`
- `Contributor` (for ACR — covers ACR mgmt plane)
- `Monitoring Contributor`

**Plus at resource scope:**
- `Key Vault Secrets Officer` → on the Key Vault
- `AcrPush` → on the Container Registry

> ⚠️ Since `Contributor` is required for ACR provisioning at the RG level, it subsumes several of the above management-plane roles (`Network Contributor`, `Storage Account Contributor`, `Key Vault Contributor`, etc.). In practice, assigning `Contributor` at the RG scope + `Key Vault Secrets Officer` at the Key Vault + `AcrPush` at the Registry is the pragmatic minimum. However, if ACR is pre-provisioned separately, you can avoid `Contributor` entirely and use only the narrower roles listed above.

## Principal 2 — Application Managed Identity

> Runtime access only — reads secrets, reads/writes blobs, enqueues/dequeues queue messages, queries SQL.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Azure Key Vault | `Key Vault Secrets User` | Key Vault | Data | Read-only access to secret values. Minimum role for application runtime. |
| Azure Storage (Blob) | `Storage Blob Data Contributor` | Storage Account or Container | Data | Read and write blobs (media uploads). Scope to specific container if possible. |
| Azure Storage (Queue) | `Storage Queue Data Contributor` | Storage Account or Queue | Data | Send, receive, peek, and delete queue messages. See note below for narrower alternative. |
| Azure SQL Database | *Not Azure RBAC* — use Entra ID contained database user | Database | Data (SQL) | See Entra ID authentication section below. |

### Narrower Queue Alternative

If the application has separate producer and consumer components:

| Component | Role | Purpose |
|---|---|---|
| Producer (enqueue only) | `Storage Queue Data Message Sender` | Write-only — minimum for message producers |
| Consumer (dequeue only) | `Storage Queue Data Message Processor` | Peek, receive, and delete messages — minimum for consumers |

### Azure SQL — Entra ID Authentication (Data Plane)

Azure RBAC does **not** grant SQL data access. Configure Entra ID authentication:

```sql
-- Run as Entra ID admin on the SQL server:
CREATE USER [app-managed-identity-name] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [app-managed-identity-name];
ALTER ROLE db_datawriter ADD MEMBER [app-managed-identity-name];
```

This grants read/write access to SQL data. For query-only (read), assign only `db_datareader`.

## Principal 3 — Developer (Read-Only)

> View resource configurations and read application logs. Must NOT modify production resources or access secret values.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| All resources | `Reader` | Resource Group | Management | View-only access to all resource configurations, settings, and metadata. |
| Application Insights / Log Analytics | `Monitoring Reader` | Resource Group | Management | View metrics, dashboards, and alert states. |
| Log Analytics (query logs) | `Log Analytics Reader` | Log Analytics Workspace | Data | Query application logs, diagnostic logs, and saved searches. Required for reading App Service logs shipped to Log Analytics. |
| Azure Key Vault | — (no role) | — | — | **Do NOT assign** `Key Vault Secrets User` or `Key Vault Reader` with data access. `Reader` on the RG shows vault metadata only — secret values remain inaccessible. |
| Azure SQL Database | — (no role) | — | — | No Azure RBAC or SQL data-plane role. `Reader` on the RG shows database metadata only. |
| App Service (live log stream) | — (not granted) | — | — | Live Log Stream requires `Website Contributor`, which grants modification rights. Use Log Analytics queries instead for read-only log access. |

> **Key restriction**: The `Reader` role shows resource metadata (SKU, config, network rules) but never exposes data-plane content (secret values, blob data, SQL data, queue messages).

## Principal 4 — Database Administrator

> Full control of Azure SQL Database (server + databases). No access to any other resource.

| Resource | Role | Scope | Plane | Notes |
|---|---|---|---|---|
| Azure SQL Logical Server | `SQL Server Contributor` | SQL Server resource | Management | Full server management: firewall rules, Entra ID admin, failover groups, auditing config. |
| Azure SQL Database | `SQL DB Contributor` | SQL Server resource | Management | Create/delete databases, change tier/SKU, configure backups and geo-replication. |
| Azure SQL (data access) | Entra ID admin on server, or `db_owner` per database | Database | Data (SQL) | Set the DBA's Entra ID group as the server Entra ID admin for full T-SQL control. |
| Azure SQL (security) | `SQL Security Manager` | SQL Server resource | Management | Optional — enables Microsoft Defender for SQL, auditing, and vulnerability assessment. |
| All other resources | — (no role) | — | — | No role assignments on VNet, Storage, Key Vault, ACR, App Service, LB, or App Gateway. |

> **Scope discipline**: Assign `SQL Server Contributor` and `SQL DB Contributor` on the **SQL Server resource** only — not the Resource Group — to prevent any visibility into other resources.

## Management Plane vs. Data Plane Summary

| Resource | Management Plane Role | Data Plane Role | Separation |
|---|---|---|---|
| Azure Storage Account | `Storage Account Contributor` | `Storage Blob Data Contributor`, `Storage Queue Data Contributor` | ✅ Fully separated — mgmt role does NOT grant data access |
| Azure Key Vault | `Key Vault Contributor` | `Key Vault Secrets User` / `Key Vault Secrets Officer` | ✅ Fully separated — mgmt role does NOT grant secret/key access |
| Azure SQL Database | `SQL DB Contributor` / `SQL Server Contributor` | Entra ID contained users (T-SQL grants) | ✅ Fully separated — Azure RBAC grants zero SQL data access |
| Azure Container Registry | `Contributor` | `AcrPush` / `AcrPull` | ✅ Fully separated — mgmt role does NOT grant image push/pull |
| App Service | `Website Contributor` | N/A (code deployment is via mgmt plane) | ⚠️ `Website Contributor` covers both config and deployment |
| VNet / NSG / LB / App GW | `Network Contributor` | N/A (no data plane) | — All management plane |

## Key Points

- **Never use storage account keys or SQL admin passwords** — assign Entra ID data-plane roles to managed identities and use Entra ID authentication exclusively.
- **`Storage Account Contributor` can list account keys**, which provides full data-plane-equivalent access — assign it only to the CI/CD SP, never to application identities.
- **`SQL DB Contributor` grants zero SQL data access** — this is the most common misconception. Data access requires T-SQL grants via Entra ID contained users.
- **Scope to the narrowest level**: CI/CD SP roles at Resource Group, application MI roles at individual resources (Key Vault, Container, Queue), DBA roles at the SQL Server resource only.
- **Disable the ACR admin account** (`adminUserEnabled: false`) — use `AcrPush` / `AcrPull` with Entra ID identities for all image operations.
- **Key Vault References** (`@Microsoft.KeyVault(SecretUri=...)`) in App Service settings are preferred over storing secrets in app settings — the app's managed identity needs only `Key Vault Secrets User`.
